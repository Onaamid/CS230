The client was The Gaming Room, and they wanted to expand their existing game, Draw It or Lose It, from an Android-only application into a web-based game that could support multiple platforms. The software requirements centered on supporting multiple games with one or more teams per game and multiple players per team, while enforcing unique names for games and teams so users could confirm whether a name was already in use. The design also needed to ensure there was one authoritative service instance in memory to manage creation, lookups, and unique identifiers for games, teams, and players. One thing I did particularly well in the documentation was translating these requirements into clear design constraints and tying them back to the domain model, so it was obvious how the classes and relationships supported what the client needed. Working through the design document was also helpful because it forced me to think through structure and tradeoffs before writing code. Having the requirements, constraints, and domain model written out made implementation more organized and reduced guesswork, especially when adding rules like unique naming and centralized management.

If I could revise one part of my work, I would improve the evaluation and recommendations sections by making them even more concise and more directly tied to the client’s priorities, such as scaling to many users and maintaining consistent behavior across platforms. I would also add slightly clearer tradeoff explanations, like how supporting many browsers and mobile devices increases testing needs, or why certain hosting approaches are typically preferred when scalability and cost control matter. Overall, I interpreted the user’s needs by focusing on what would matter in real use: players need a smooth experience across devices, and the business needs a structure that prevents confusion (duplicate names) and supports growth. Considering user needs is important because it keeps the design grounded in how the system will actually be used, not just what is easiest to code. My approach to designing the software was to start with requirements, translate them into constraints, and then build a domain model that reflected the relationships in the system before implementation. In the future, I would use the same strategy and strengthen it by writing clearer use cases early, defining scope boundaries, and planning test scenarios up front so cross-platform behavior, uniqueness checks, and access rules can be validated as the system is developed.
